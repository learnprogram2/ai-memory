# PR Description Command

Generate a well-structured, detailed pull request description for the current branch against `origin/master`.

## Instructions

1. Run `git log --oneline origin/master..HEAD` to understand the commit history
2. Run `git diff origin/master...HEAD` to get the full diff
3. Analyze all changes thoroughly — every file, every meaningful modification
4. Write the PR description following the repo's template format below

## Format

```markdown
## Description

A concise paragraph (2-4 sentences) explaining:
- **What** this PR does (the feature/fix/change)
- **Why** it exists (the motivation, what system or workflow triggers it)
- **How** it works at a high level (key design decisions, architectural choices)

Then group changes by area with clear headers. For each area, use bullet points describing **what changed and why**. Be specific — mention types, field names, config keys, status values, endpoint paths, etc.

Common area headers (use only what applies):
- **Protobuf / API** — new/modified proto definitions, gRPC/Connect endpoints, request/response schemas
- **Domain** — business logic, service interfaces, repository patterns, new domain types
- **Adapters** — storage implementations, external service integrations
- **Database** — migrations, queries (goyesql), new columns/indexes, constraints
- **Temporal Workflows** — new/modified workflows, activities, signals, queries
- **Configuration** — TOML config, Viper bindings, environment variable mappings
- **Tests** — what's covered, how many tests, key scenarios
- **Infrastructure** — Dockerfile, Makefile, CI/CD, dependencies

## Types of changes:

<!-- Check all that apply based on the diff analysis -->
- [ ] Bug fix.
- [ ] Technical debt.
- [ ] New feature.
- [ ] Breaking change (Devops, please be aware of).

## Merge request checklist:

- [ ] Label your MR for auditing using GitHub labels `major`, `minor`, `patch`, `security patch`
    - `major` - backward incompatible or breaking changes
    - `minor` - when you add functionality in a backward-compatible manner
    - `patch` - when you make backward-compatible bug fixes
- [ ] My code follows the [Luxor Go Development Guidelines](https://www.notion.so/luxorlabs/Go-development-guidelines-b19a05f6f35e4c8ebe3bea2fc6af93ae).
- [ ] My commits follow the [How to Write a Git Commit Message Guide](https://chris.beams.io/posts/git-commit/).
- [ ] I have updated the [CHANGELOG](./CHANGELOG.md) (yeah, last chance to do it).
```

## Rules

- Be detailed and specific — mention actual field names, types, config keys, query names, activity names
- Don't be vague ("updated some files") — say exactly what changed
- If there are important design decisions (e.g., idempotency strategies, retry policies, state machines), explain them
- For protobuf/API changes, list the service methods, request/response shapes, and key validations
- For Temporal workflows, describe the workflow structure, activity sequence, and timeout/retry configuration
- For database changes, mention column types, nullable/defaults, index types, and query semantics
- For tests, summarize coverage areas (happy path, error cases, edge cases) with counts
- In the "Types of changes" section, check (`[x]`) the boxes that apply based on the actual changes in the diff
- Do NOT include a test plan section
- Do NOT include "Generated by Claude" or any AI attribution
- Do NOT include PR title — just the body
- Wrap the ENTIRE output in a single markdown code fence (triple backticks with `markdown` language tag) so the raw markdown is visible and easy to copy from the terminal
- No commentary, no preamble — just the code-fenced PR description
